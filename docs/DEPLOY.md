# PFMS Deployment Guide

This document walks through three supported deployment targets in detail.

1. [Self-hosted Docker](#1-self-hosted-docker) — recommended for personal use.
2. [Render.com (free tier)](#2-rendercom-free-tier) — easiest managed hosting.
3. [Fly.io (free tier)](#3-flyio-free-tier) — global edge deploy, generous free CPU.

All three targets use the same backend Docker image, with deployment-specific
configuration provided via environment variables.

> Common prerequisites
>
> * A PostgreSQL database, version 14+.
> * A Z.ai API key for receipt vision (set `ZAI_API_KEY`). The default
>   provider is Z.ai's GLM-5V; Anthropic Claude Sonnet is also supported
>   (`ANTHROPIC_API_KEY`) and can be selected with `VISION_PROVIDER=anthropic`.
> * A strong `JWT_SECRET`. Generate with:
>
>   ```bash
>   openssl rand -hex 64
>   ```
>
>   The backend hard-fails in production if `JWT_SECRET` is missing, a known
>   weak default, or shorter than 32 characters.

## 1. Self-hosted Docker

**Best for:** running PFMS on your own server or laptop.

### 1.1 Prerequisites

* Docker Engine 20.10+ and Docker Compose v2.
* (Optional) A reverse proxy / TLS terminator (Caddy, Traefik, nginx).

### 1.2 First-time setup

```bash
git clone <your-fork-url> pfms
cd pfms

cp .env.example .env
# Edit .env — fill in at minimum:
#   JWT_SECRET=<openssl rand -hex 64>
#   ZAI_API_KEY=zai-...
#   CORS_ORIGIN=http://localhost
#   POSTGRES_PASSWORD=<something strong>
```

### 1.3 Bring everything up

```bash
docker compose up -d --build
```

What this does:

1. Starts `postgres` (port `5433` on host).
2. Runs the one-shot `migrate` service. It applies every `backend/migrations/*.sql`
   file not yet recorded in `schema_migrations` and exits.
3. Starts `backend` (port `5000`).
4. Starts `frontend` (port `80`).

Healthchecks gate dependencies: the API does not start until the DB is healthy
**and** migrations have completed.

### 1.4 Verify

```bash
docker compose ps           # all services healthy
curl http://localhost/      # frontend served by nginx
curl http://localhost:5000/api/health
```

Then open `http://localhost` in a browser and register an account.

### 1.5 Day-2 operations

| Task | Command |
|---|---|
| Apply new migrations | `docker compose run --rm migrate` |
| Tail backend logs | `docker compose logs -f backend` |
| Backup database | `docker compose exec postgres pg_dump -U pfms pfms > backup.sql` |
| Restore database | `cat backup.sql \| docker compose exec -T postgres psql -U pfms pfms` |
| Update | `git pull && docker compose up -d --build` |

### 1.6 Putting it behind TLS

If you expose PFMS publicly, terminate TLS at a reverse proxy. Example with Caddy:

```
pfms.example.com {
  reverse_proxy /api/* localhost:5000
  reverse_proxy /uploads/* localhost:5000
  reverse_proxy /* localhost:80
}
```

Set `CORS_ORIGIN=https://pfms.example.com` in `.env` and `docker compose up -d`.

## 2. Render.com (free tier)

**Best for:** the easiest managed deploy with a free Postgres instance.

> Render's free Postgres expires after 90 days. For production use, upgrade to a
> paid plan or schedule nightly backups (see §2.6).

### 2.1 Sign up + connect repo

1. Create an account at <https://render.com>.
2. Push your fork to GitHub/GitLab.
3. From the Render dashboard: **New + → Blueprint**.
4. Select the repo containing `render.yaml`.
5. Confirm the resources Render proposes: `pfms-backend`, `pfms-frontend`,
   `pfms-db`. Click **Apply**.

### 2.2 Set secrets

In the Render dashboard:

1. Open the **pfms-backend** service → **Environment**.
2. Set:
   * `ZAI_API_KEY` → your Z.ai key.
   * `ANTHROPIC_API_KEY` → optional fallback.
3. `JWT_SECRET` is generated by Render automatically (`generateValue: true`).
4. After the frontend service deploys, copy its URL (e.g.
   `https://pfms-frontend.onrender.com`) and set
   `CORS_ORIGIN` on the backend to that URL.
5. Save and trigger a manual redeploy of the backend.

### 2.3 Migrations

`render.yaml` declares a `preDeployCommand: node dist/models/runMigrations.js`.
Render runs this once per deploy *before* swapping in the new container, so
two replicas never race on schema changes. You don't have to do anything
manually for normal deploys.

To run migrations ad-hoc:

```bash
render shell pfms-backend
node dist/models/runMigrations.js
```

### 2.4 Uploads

Render free web services have an **ephemeral filesystem** — uploaded receipts
vanish on redeploy. Two options:

* **Disk add-on** (paid): attach a persistent disk and point
  `UPLOAD_DIR` at the mount path.
* **Object storage**: configure S3 / Cloudflare R2 (planned in a later phase).

Until then, treat self-hosted Docker or paid Render as the durable option for
receipt history.

### 2.5 Custom domain

In the Render dashboard:

1. **pfms-frontend → Settings → Custom Domains → Add.**
2. Add your CNAME at your DNS provider as instructed.
3. Update the backend's `CORS_ORIGIN` to match the new origin.

### 2.6 Backups

Free Postgres on Render does not auto-back-up.

```bash
# Run from your laptop, with `DATABASE_URL` set to the External Database URL
# shown on the Render database page.
pg_dump "$DATABASE_URL" > "pfms-$(date +%F).sql"
```

Schedule the above as a daily cron on any host you own.

## 3. Fly.io (free tier)

**Best for:** global low-latency deploys; free for small workloads.

### 3.1 Install flyctl + sign in

```bash
# https://fly.io/docs/hands-on/install-flyctl/
fly auth signup     # or: fly auth login
```

### 3.2 Create the app and Postgres

```bash
# In repo root, where fly.toml lives
fly apps create pfms                 # name must be globally unique
fly postgres create --name pfms-db   # accept default region; pick the smallest plan
fly postgres attach --app pfms pfms-db
```

`attach` injects `DATABASE_URL` into the app's secrets.

### 3.3 Set application secrets

```bash
fly secrets set --app pfms \
  JWT_SECRET="$(openssl rand -hex 64)" \
  ZAI_API_KEY="zai-..." \
  CORS_ORIGIN="https://pfms.fly.dev"
```

Optionally:

```bash
fly secrets set --app pfms ANTHROPIC_API_KEY="sk-ant-..."
fly secrets set --app pfms VISION_PROVIDER="zai"
```

### 3.4 Deploy

```bash
fly deploy
```

The Dockerfile is a multi-stage build, so the image stays small (~150 MB).

`fly.toml` sets `min_machines_running = 1` so the background scheduler runs.
You can lower to `0` if you don't need recurring-transaction detection.

### 3.5 Run migrations

`RUN_MIGRATIONS_ON_BOOT=false` is set in `fly.toml`, so migrations are a manual
step you control:

```bash
fly ssh console --app pfms -C "node dist/models/runMigrations.js"
```

Run this after every deploy that includes a new `backend/migrations/*.sql`
file. The `schema_migrations` table guards against double application, and the
advisory lock prevents two replicas from racing.

### 3.6 Verify

```bash
curl https://pfms.fly.dev/api/health
fly logs --app pfms
```

### 3.7 Persistent uploads

Fly machines have ephemeral root filesystems. Attach a Volume:

```bash
fly volumes create pfms_uploads --app pfms --size 1 --region iad
```

Then in `fly.toml`:

```toml
[mounts]
  source = "pfms_uploads"
  destination = "/app/uploads"
```

…and redeploy.

### 3.8 Backups

```bash
# One-off
fly postgres connect --app pfms-db
# In psql: \! pg_dump > /tmp/dump.sql

# Cron-friendly
fly postgres backup list --app pfms-db
fly postgres backup create --app pfms-db
```

## Common: post-deploy sanity checklist

After any deploy, verify:

```bash
# Health
curl https://<host>/api/health

# Auth flow works
curl -X POST https://<host>/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"you@example.com","password":"StrongPass123","name":"Me"}'

# JWT secret hardening triggers in production
# (this is informational — a misconfigured server will fail to boot loudly)
```

Then in the UI:

1. Register, log in.
2. Create an account.
3. Upload a receipt — verify the scan completes and a transaction is suggested.
4. View **Analytics** — confirm the Net Worth chart and Cash-flow Forecast
   chart render.

If any step fails, check `fly logs` / Render logs / `docker compose logs backend`.

## Environment variable reference

| Variable | Default | Purpose |
|---|---|---|
| `NODE_ENV` | `development` | Toggles strict checks (e.g., JWT_SECRET hardening). |
| `PORT` | `5000` | API listen port. |
| `DATABASE_URL` | local pg | Postgres connection string. |
| `JWT_SECRET` | required in prod | Used to sign access tokens. ≥32 chars. |
| `JWT_EXPIRES_IN` | `7d` | Access-token lifetime. Consider `15m` once refresh tokens are used. |
| `JWT_REFRESH_EXPIRES_IN` | `30d` | Refresh-token lifetime. |
| `ZAI_API_KEY` | empty | Z.ai GLM-5V API key (default vision provider). |
| `ANTHROPIC_API_KEY` | empty | Optional Claude Sonnet fallback. |
| `VISION_PROVIDER` | `zai` | `zai` or `anthropic`. |
| `UPLOAD_DIR` | `./uploads` | Local directory for receipt uploads. |
| `MAX_FILE_SIZE_MB` | `20` | Per-file upload cap. |
| `SCANS_PER_USER_PER_DAY` | `50` | Daily quota on vision-API spend. |
| `CORS_ORIGIN` | `http://localhost:5173` | Comma-separated allowlist of allowed origins. |
| `RUN_MIGRATIONS_ON_BOOT` | `false` in prod | Set to `true` to apply migrations at boot (dev only). |
| `LOG_LEVEL` | `info` / `debug` | pino log level. |
